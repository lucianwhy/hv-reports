Agent 支付研究报告

从 HTTP 402、MCP、x402、AP2、ACP 到 AgentCore Payments，理解 Agent 支付如何从概念进入基础设施成形期。

一句话定义

Agent 支付，是指 AI Agent 在明确授权、预算约束、可审计日志和可验证身份前提下，代表个人或企业完成资源发现、价格确认、支付执行、结果获取和账证留存的机器化支付流程。

摘要

过去谈支付，默认主体一直是人。无论是银行卡、钱包、网银还是第三方支付，系统假设的都是一个人正在看页面、理解价格、点击确认、承担结果。Agent 支付改变的恰恰是这个基本前提。它不是给传统支付流程外面包一层 AI，也不是让模型学会填写结账表单，而是把支付重新放回机器决策链条里，让发现可付费资源、判断是否值得支付、在限制条件下完成支付并继续执行任务，变成一种原生的系统能力。

2025 到 2026 年，这个方向开始明显从概念讨论进入基础设施成形期。一头是 MCP 之类的工具协议，让 Agent 调用外部服务从零散技巧变成标准能力；另一头是 x402、AP2、ACP、UCP、AgentCore Payments 等支付与交易协议或平台层能力，开始把授权、结算、审计、商户接入、预算治理这些问题拉进统一讨论。它们共同说明一件事：未来互联网里，支付不再只是人类消费的末端动作，而会成为机器消费资源、调用服务、完成商业交易的一段关键中间层。

如果说上一代互联网的问题是信息如何自由流动，那 Agent 互联网逼近的新问题，就是价值如何在机器之间低摩擦、可验证地流动。

纵向分析

Agent支付的纵向演化历史，若要追溯其最早的技术预言，必须从一个几乎被遗忘的HTTP状态码说起。1996年，HTTP/1.1规范草案中出现了402状态码，其官方名称为“Payment Required”，设计初衷是预留给未来的微支付基础设施使用。彼时互联网商业化方兴未艾，设计者已经隐约预见到机器与机器之间、或用户与服务之间存在自动结算的需求。然而这个状态码在此后将近三十年的时间里始终处于“保留未用”的状态，既没有形成配套协议标准，也没有获得主流平台的实现支持。其落地失败的根本原因是多层次的：一方面，传统网页的消费模式以人为中心，支付行为需要明确的用户意图与授权界面，嵌入HTTP层的无感支付在彼时缺乏实际场景；另一方面，微支付在技术经济层面长期面临交易手续费与交易金额倒挂的困境，信用卡网络的最低费率结构使得分币级别的结算在经济上不可行；此外，身份认证、欺诈风险与监管合规等问题也使得任何试图自动化支付的尝试都难以形成闭环。402因此成为互联网协议史上最著名的占位符，承载着一种超前于时代的构想，却始终等待着正确的基础设施与应用场景来激活它。

进入大型语言模型时代后，这一等待开始有了答案的轮廓。2024年前后，以 Anthropic 为核心推动的 MCP 逐渐确立为 Agent 工具调用的事实标准之一。MCP 的核心价值在于将外部工具、数据源与 API 接口以标准化方式暴露给语言模型，使 Agent 能够在推理过程中动态调用能力。这一架构变化带来了一个此前不存在的现实需求：当 Agent 调用一个外部工具时，该工具可能是商业服务，需要按次或按量计费。传统的 API 密钥预付费模式假设背后有一个人类开发者事先充值、事先授权，但在完全自主运行的 Agent 场景下，推理链条可能在运行时才决定调用哪些工具，静态的预授权机制既粗糙又低效。MCP 让付费工具调用从一个理论假设变成了大量开发者日常面对的工程问题，也因此催生了对 Agent 原生支付层的真实需求。

2025 年，这一领域迎来了密集的基础设施建设期。从协议层看，x402 的出现是一个具有转折意义的节点。它直接复活了 HTTP 402 的语义，将其实现为一套完整的机器可读支付握手协议：当服务端需要收费时，返回 402 响应并在头部携带支付要求的结构化描述，客户端解析后可自动完成支付并重放原始请求。这一设计的深刻之处在于它将支付行为内嵌于 HTTP 交互本身，无需人类介入、无需跳转界面、无需预先建立双边账户关系。x402 以稳定币作为初期结算层，绕开了传统支付网络的最低费率壁垒，使分厘级别的即时结算在技术经济上首次变得可行。更重要的是，x402 将支付能力下沉为 HTTP 协议的扩展而非应用层的特殊逻辑，这意味着任何兼容 HTTP 的 Agent 框架理论上都能以极低的集成成本接入这套机制。

与此同时，OpenAI 与 Stripe 围绕 agentic commerce 的合作方向，以及 ACP 框架，揭示了另一条演化路径。这条路径的逻辑起点是：大量消费者与企业已在既有支付基础设施中存有支付凭据，若能将这些凭据以安全、受限的方式授权给 Agent 使用，则可在现有法币支付体系上构建 Agent 支付能力，而无需强制引入加密资产。ACP 方向试图定义 Agent 之间委托授权的标准，包括资金使用范围、单次限额、有效期等细粒度权限声明，使用户可以像授权 OAuth 应用一样向 Agent 授予有界的支付权限。这一路径的优势在于合规性更强、用户心理接受门槛更低、与现有企业财务系统的整合更自然；其挑战则在于传统法币结算的延迟与手续费结构，在超高频微支付场景下依然构成摩擦。两条路径，即基于稳定币的链上即时结算与基于既有支付凭据的授权委托，并非完全对立，而是在不同场景与不同用户群体中形成互补。

Google 在 2025 年 9 月发布的 AP2，则代表头部平台将 Agent 支付能力纳入自身生态标准化体系的战略动作。AP2 的发布意义不在于技术细节上的绝对创新，而在于它释放的信号：全球头部平台正式承认 Agent 自主发起支付是需要被协议层规范的一类行为。AP2 进一步推动了支付能力在 Agent 基础设施层面的标准化进程，也为后续不同 Agent 框架、不同支付提供商之间的互操作性奠定了讨论基础。更深远的影响在于，AP2 将 Agent 工作流与支付、身份和商户基础设施打通，为企业级 Agent 部署提供了一个可审计、可追溯、符合监管预期的支付路径。

如果说 x402 与 AP2 解决的是协议与标准层面的问题，AWS AgentCore Payments 的出现则标志着 Agent 支付演化进入了一个新的阶段，即企业托管阶段。AgentCore 作为 AWS 面向企业级 Agent 部署的托管平台，其 Payments 模块将支付能力作为平台原生服务提供，企业无需自行集成支付 SDK、无需维护密钥管理逻辑、无需单独应对合规审计，而是通过 IAM 权限体系将支付操作与其他云资源的访问控制统一管理。这一模式的战略含义在于：它将 Agent 支付的信任锚点从协议层转移到了云平台层，企业的风控、审计与财务团队可以用他们熟悉的云账单与权限管理工具来监管 Agent 的支付行为。AgentCore Payments 意味着支付能力不再是 Agent 应用需要外挂解决的边缘问题，而是成为企业级 Agent 运行时基础设施的标准组件。

回溯这整条演化线索，可以发现一个清晰的递进逻辑：从 HTTP 402 的协议预言，到 MCP 将付费工具调用变为真实工程需求，到 x402 以稳定币完成对 402 语义的首次真实激活，到 Google AP2 与 OpenAI / Stripe 路径的并行推进确立标准化竞争格局，再到 AWS AgentCore Payments 将企业托管能力打包交付，每一步都在扩大 Agent 自主支付的可用范围，同时降低下一个参与者的准入门槛。这个演化过程的本质，是一种信任传递机制的逐步成熟：从要求人类实时授权，到允许人类事先设定有界权限，再到平台以托管方式承担合规担保。Agent 支付的历史，在某种意义上正是人类对机器自主行动的信任边界被技术与制度共同推进的历史。

关键时间轴

  1. 1996 年：HTTP 402 被放入协议草案，提出 Payment Required 语义，但没有现实落地条件。
  2. 2024 年前后：MCP 等工具协议让 Agent 调用外部服务逐步标准化，付费工具调用需求浮出水面。
  3. 2025 年：x402 通过稳定币与 HTTP 协议结合，让机器原生微支付第一次具备工程可行性。
  4. 2025 年 9 月：Google 发布 AP2，将支付问题正式拉入 Agent 协议栈讨论。
  5. 2025 年下半年：OpenAI 与 Stripe 沿着 agentic commerce / ACP 路线推进，商户侧 Agent 支付框架成形。
  6. 2026 年：AWS 推出 AgentCore Payments，Agent 支付进入企业托管与生产治理阶段。

横向分析

Agent 支付的横向竞争并不是单一“支付按钮”之争，而是围绕智能体在不同交易场景中如何获得授权、如何表达意图、如何完成结算、如何承担风险与合规责任的体系性分化。当前几条主要路线分别从协议、云平台、商户结账、跨主体编排和支付网络信任层切入，解决的是同一问题链条中的不同环节，因此既存在竞争，也存在明显的互补关系。总体来看，x402 更偏向互联网原生的机器对机器微支付，AWS AgentCore Payments 更偏向企业内部与云上工作流的托管支付能力，Stripe 与 ACP 更偏向面向商户的结账转化与收单效率，Google AP2 / UCP 更强调智能体交易中的意图、授权与多方协议编排，而 Visa、Mastercard、PayPal 则试图将既有支付网络、风控、消费者保护和商户覆盖能力上移为智能体时代的信任基础设施。

x402 协议型微支付路线解决的核心问题，是如何让智能体在访问 API、数据、内容、算力或数字服务时，以极低摩擦完成小额、即时、程序化支付。传统 HTTP 世界缺少原生支付语义，导致机器请求与价值交换分离，开发者往往要额外集成账户体系、订阅系统或第三方结账流程。x402 的吸引力在于把支付嵌入请求响应过程，使付费访问成为协议层能力，特别适合 API 调用计费、AI 工具调用、数据流订阅、内容按次访问、模型推理微付费等场景。其优势是轻量、开放、可组合，符合智能体高频、小额、自动化交互的特征。但短板同样明显：微支付协议本身难以独立解决 KYC、AML、退款、争议处理、税务、消费者保护和商户准入等现实问题；如果依赖加密资产或稳定币结算，还会面临监管、波动性、钱包体验和法币出入金问题。因此，x402 更像是智能体经济中的价值交换语法，而不是完整支付产业栈。

AWS AgentCore Payments 的企业托管路线则从另一个方向切入：它解决的不是开放互联网中任意智能体之间如何付款，而是企业在云环境中如何安全、可控、可审计地让智能体代表组织执行采购、调用服务或触发支付。企业采用智能体最大的顾虑并非能否完成一次支付，而是权限边界、预算控制、审批流、凭证管理、审计追踪和合规责任如何落地。AWS 的优势在于其云基础设施、企业客户基础、身份权限管理、日志监控和工作流编排能力，可以把支付能力封装进企业级智能体运行环境中。它适合云上采购、企业 SaaS 调用、内部自动化运维、供应链补货、云资源消费和 B2B 服务结算等场景。短板是开放性和跨生态普适性可能受限，企业托管模式天然依附于云平台边界，容易形成平台锁定。

Stripe / ACP 商户结账路线更贴近现有商业互联网的现实需求。它主要解决智能体替用户或企业在商户侧完成购买时，如何延续成熟的收单、结账、风控、订阅、发票、税务和退款能力。对大量商户而言，他们并不希望为了 Agent 支付重建一套收款系统，而是希望智能体能够像人类用户一样进入既有结账流程，同时更高效地提交购买意图、配送信息、优惠信息和支付凭证。Stripe 的优势在于开发者体验、商户覆盖、支付方式聚合和商业化工具链；ACP 类路线若能标准化智能体与商户之间的结账交互，就可以降低代理购买中的信息不完整、表单填写失败、身份误判和交易风控误伤。它适合电商、订阅软件、在线服务、本地服务预约、旅行预订等有明确商户和售后责任的场景。短板在于它仍然围绕商户结账页面或商户支付栈展开，对机器之间的微支付、跨多个服务的复合任务、企业审批链或开放协议互操作支持有限。

Google AP2 / UCP 协议编排路线关注的是智能体交易中最复杂也最容易被忽视的问题：智能体如何证明自己得到了用户授权，如何表达可验证的购买意图，如何在多个参与方之间传递约束条件，并在最终付款前保留可追溯的决策链。相比单纯支付，AP2 / UCP 更像交易前后的协议层与编排层，试图把谁授权了什么、在什么条件下、由哪个智能体执行、结果如何确认标准化。它适合跨平台搜索、比价、预订、组合购买、多智能体协作、开放网页代理购物等场景。其优势是视野更高，试图为 Agent commerce 建立通用语义和可验证流程；短板是落地依赖生态共识，标准推进周期长，且协议本身并不天然拥有商户受理网络、资金清算能力或消费者信任品牌。

Visa、Mastercard、PayPal 的网络级信任路线则从既有金融网络向上延伸，核心解决的是智能体支付在大规模商业化时最关键的信任与责任分配问题。智能体代用户下单会带来新的欺诈形态、授权争议、误购退款、商户责任、账户接管和身份冒用风险。卡组织和大型钱包的优势在于全球受理网络、发卡与收单关系、争议处理规则、令牌化、风控模型、消费者保护和品牌信任。它们适合主流零售、电商、跨境消费、旅行、本地服务以及高价值交易等需要强风控和售后保障的场景。相比 x402 的轻量开放，它们更重、更合规、更接近金融监管体系；相比 Stripe 的商户侧工具，它们处在更上游的网络规则层；相比 Google 的协议编排，它们掌握真实支付授权和清算网络。短板是创新速度可能较慢，规则体系复杂，成本相对较高，对开放微支付和长尾开发者场景不够友好。

从彼此关系看，这些路线并非简单替代，而是在不同层级争夺智能体支付的控制点。x402 争夺的是机器请求中的支付原语，适合作为底层开放接口；AWS 争夺的是企业智能体运行环境中的治理入口；Stripe / ACP 争夺的是商户结账与收单转化入口；Google AP2 / UCP 争夺的是跨主体交易意图和授权表达的标准；Visa、Mastercard、PayPal 争夺的是最终资金流、风险承担和消费者信任网络。未来更可能出现分层耦合而非赢家通吃：一个企业智能体可能运行在 AWS 上，按照 AP2 表达授权意图，通过 Stripe 商户接口完成结账，底层使用 Visa 令牌化凭证完成授权；而在 API 调用和数字资源访问中，则可能采用 x402 式微支付协议，并由稳定币、钱包或传统支付网络提供结算支持。真正的竞争焦点在于谁能成为默认入口和规则制定者。若交易是低额、高频、数字原生，协议型微支付更有优势；若交易是企业可控流程，云托管路线更具黏性；若交易面对现有商户，商户结账路线最现实；若交易跨平台、多方协作，协议编排路线不可或缺；若交易进入大众消费和高价值支付，网络级信任仍是最终基础。由此判断，Agent 支付不会沿单一路线演进，而会形成协议层开放化、平台层托管化、商户层标准化、网络层信任化的多轨并行格局。

横纵交汇洞察

随着人工智能从信息交互向经济行为主体的跃迁，智能体支付正在重塑现有的金融基础设施。在这个演进过程中，预算控制、授权边界与审计日志构成了智能体支付设计的核心三要素。不同于人类支付依赖视觉交互界面和单次意图确认，智能体支付具有连续性、高频次和自主性，这使得传统安全校验机制面临失效风险。预算控制是防御大模型幻觉或逻辑死循环导致资金流失的最后防线，它要求系统能够支持基于任务维度、时间周期或特定触发条件的精细化资金分配，而非简单的账户余额消耗。授权边界则定义了智能体的商业行为半径，通过可编程的白名单或智能合约，严格限定智能体只能在特定的商户网络、服务接口或资产类别中进行交易，从而将未知的欺诈风险隔离在系统之外。同时，审计日志在智能体支付中不再仅仅是财务对账的工具，而是机器行为可解释性的基础。详尽且不可篡改的机器级交易日志不仅用于事后追责与纠纷仲裁，更是用于反向微调和强化学习的重要数据，确保智能体的经济决策模型能够在迭代中不断逼近人类委托者的真实意图。

在基础设施的演进路线上，市场曾有一种假设，认为最终会出现一个统一的通用协议吃掉所有的智能体支付场景，但这在商业现实和物理限制面前是不成立的。智能体面临的商业场景具有极高的离散度，从毫秒级的高频微交易到复杂的企业级供应链采购，其对结算延迟、隐私保护、合规成本和网络外部性的要求存在不可调和的矛盾。没有任何单一协议能够同时满足去中心化的机器信任、海量并发的吞吐能力以及与传统实体经济的无缝对接。因此，未来的智能体支付网络注定是多层且异构的。

稳定币凭借其无边界、可编程和即时结算的特性，将在原生的机器对机器经济中占据优势，尤其适用于跨国界的接口调用付费、算力租赁以及高频微交易网络。然而，当智能体需要跨越数字世界向物理世界采购服务时，传统卡网络仍将是不可或缺的桥梁。其庞大的商户接受度以及成熟的争议解决和拒付机制，使得卡网络在智能体代客预订机票、采购实体商品或订阅传统软件等场景中具有压倒性优势，因为此类场景的核心痛点不是支付效率，而是履约信任。相比之下，托管钱包将在企业级内部智能体生态和高度合规的闭环平台中发挥关键作用。大型企业更看重合规性、税务处理的便利性以及与现有系统的无缝整合，托管钱包能够提供更流畅的法币通道转换和更低的基础设施维护摩擦，牺牲部分去中心化特性以换取企业级所需的绝对控制权和责任主体明确性。

评估智能体支付是否真正跨越鸿沟，不能仅凭技术叙事，而需要严密的商业指标验证。在未来十二到二十四个月的关键窗口期，有几个核心验证指标最值得市场关注。首先是完全自主交易量与人类参与验证交易量的比例变化，这一指标直接反映了系统信任度的爬坡曲线，只有当智能体在没有人类介入的情况下完成的交易比例出现非线性增长，才意味着预算与授权机制真正走向成熟。其次是单体智能体的平均交易频次与客单价，初期的智能体支付必然集中在低价值的高频试错场景，客单价的持续跃升将是智能体从边缘长尾任务向高附加值核心业务渗透的明确信号。此外，智能体交易的争议率和错误退款率也是衡量底层基础设施可靠性的试金石，因为理解偏差或接口适配问题导致极高的交易失败和资金回滚将直接摧毁商业模式的经济可行性。最后，传统商户端针对智能体特定结账接口的采用率同样关键，这标志着供给侧是否已经开始主动适应机器买家的崛起。

设计原则

  1. 支付必须嵌入 Agent 推理循环，而不是跳出流程另开一个人工确认世界。
  2. 授权必须可编程、可限制、可撤销，不能只靠一句模糊的允许支付。
  3. 预算与额度应该默认开启，并且最好按任务、时间和对象多维切分。
  4. 审计日志不是附属功能，而是 Agent 支付系统的基础设施。
  5. 面向数字资源的轻支付协议和面向复杂商业交易的结账协议，长期都会并存。
  6. 资源发现层、支付层、履约层和争议处理层，不应该混作一团，而应分层设计。

结论

智能体支付的本质，并非在现有金融轨道上增加一种新的支付手段，而是通过可编程的信任协议，将资本配置权首次规模化地让渡给非人类逻辑主体。它意味着商业世界正在从人发起、人点击、人承担结果的单一结构，缓慢过渡到人设定边界、机器执行交易、系统保留证据的新结构。

今天讨论 Agent 支付，已经不能再只盯着某个钱包、某个协议或某个模型是否更先进。更关键的问题是：谁来定义授权边界，谁来设置预算规则，谁来保存审计证据，谁来承担争议责任，以及谁来成为机器消费资源的默认入口。从这个意义上看，x402、AP2、ACP、AgentCore Payments、Visa、Mastercard、PayPal 这些名字背后真正竞争的，不只是功能，而是未来机器经济里的规则制定权。

Agent 支付不是给 AI 多装一个 checkout，而是在重写“机器如何合法、可信、低摩擦地消费资源”这件事本身。
